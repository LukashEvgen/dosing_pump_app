# Contributing to Dosing Pump App

Дякуємо за інтерес до проекту! Ми вітаємо контрибуції від спільноти.

## Як допомогти

### Звіти про баги

Якщо ви знайшли баг:

1. Перевірте, чи не створена вже подібна issue
2. Створіть нову issue з детальним описом:
   - Кроки для відтворення
   - Очікувана поведінка
   - Фактична поведінка
   - Версія додатку та firmware
   - Скріншоти (якщо можливо)

### Пропозиції нових фіч

1. Відкрийте issue з описом фічі
2. Поясніть, чому ця фіча корисна
3. Дочекайтесь обговорення перед початком розробки

### Pull Requests

1. **Fork** репозиторій
2. Створіть **feature branch** (`git checkout -b feature/amazing-feature`)
3. Внесіть зміни
4. **Тестування**: переконайтесь, що додаток працює
5. **Commit**: `git commit -m 'Add amazing feature'`
6. **Push**: `git push origin feature/amazing-feature`
7. Відкрийте **Pull Request**

## Стандарти коду

### Dart/Flutter

- Дотримуйтесь [Effective Dart](https://dart.dev/guides/language/effective-dart)
- Використовуйте `flutter analyze` перед commit
- Форматуйте код: `flutter format .`
- Додавайте коментарі для складних частин

### Commits

- Використовуйте чіткі commit messages
- Формат: `[type]: description`
  - `feat:` нова фіча
  - `fix:` виправлення бага
  - `docs:` зміни документації
  - `refactor:` рефакторинг коду
  - `test:` додавання тестів

Приклад: `feat: add schedule editing UI`

## Тестування

- Запустіть `flutter test` перед PR
- Додайте тести для нових фіч
- Перевірте на реальному пристрої

## API Інтеграція

При роботі з API:
- Дотримуйтесь структури, описаної у [firmware README](https://github.com/LukashEvgen/pdstepperfirmware)
- Тестуйте з реальним firmware
- Обробляйте помилки мережі

## Структура проекту

```
lib/
├── main.dart           # Точка входу
├── models/             # Моделі даних
├── services/           # API та бізнес-логіка

├── widgets/            # Переиспользуемые компоненти
└── utils/              # Утиліти
```

Нові файли мають відповідати цій структурі.

## Review Process

1. Підтверджуємо, що PR відповідає стандартам
2. Тестуємо функціонал
3. Запитуємо зміни, якщо потрібно
4. Merge після схвалення

## Питання?

- Створіть issue з міткою `question`
- Опишіть контекст вашого питання

## Code of Conduct

- Будьте ввічливими та конструктивними
- Поважайте думки інших
- Фокусуйтесь на проблемі, а не на особистостях
